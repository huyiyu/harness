## Context

`03-devops-foundation` 章节（`docs/content/docs/03-devops-foundation/_index.md`）当前为占位状态。该章节在整本书中承担"理论对接"角色——将第二章介绍的 Harness 核心思想与软件工程的经典理论（DevOps/持续交付）建立联系，证明 Harness 不是空中楼阁，而是已有成熟理念的延伸。

本次 change 为纯文档内容创作，不涉及代码变更。

## Goals / Non-Goals

**Goals:**
- 建立 Harness 与 DevOps 的理论连续性：Harness 是 DevOps 反馈环的极致延伸
- 用"反馈环速度"作为核心案例：传统 DevOps（小时级）→ Harness（分钟级）
- 说明 DevOps 基础设施是 Harness 运行的必要土壤
- 轻量提及需要重新诠释的 DevOps 实践（代码审查→Evaluator 验收、人工审批→自动门控）
- 控制篇幅在 1500-2000 中文字符，精炼不冗余

**Non-Goals:**
- 不展开 DevOps 的完整历史或理论全景（不是 DevOps 教科书）
- 不深入 SRE 或可靠性工程（留给 04 章）
- 不讨论具体工具链或配置细节（留给下篇 08-14 章）
- 不修改 Hugo 站点配置、主题或构建流程

## Decisions

**1. 叙事主线：反馈环速度**
- **选择**：用"反馈环从小时级压缩到分钟级"作为 Harness DevOps 的核心论点
- **理由**：决策者最容易感知的指标是"速度"；传统 DevOps 的瓶颈在于"人必须参与"，Harness 消除了这个瓶颈
- **替代方案**：用"自动化深度"做主线——被认为过于技术化，不如"速度"直观

**2. 结构 ACB，占比 8:1:1**
- **选择**：
  - A（80%）：Harness 是 DevOps 的终极形态——反馈环速度对比
  - C（10%）：DevOps 是 Harness 的土壤——基础设施依赖
  - B（10%）：需要重新诠释的实践——两个例子点到为止
- **理由**：一章一个核心论点，不分散火力；读者记住一个结论比记住三个结论更有效

**3. 弱化 Evaluator 验收与自动化测试的层次区分**
- **选择**：将 Evaluator 验收视为 DevOps 自动化测试的"目标驱动升级版"，而非独立层级
- **理由**：避免陷入技术架构讨论，保持管理层视角的简洁性
- **关键表述**："Evaluator 的验收不是替代自动化测试，而是把'人的判断'升级为'AI 的目标验证'"

**4. 篇幅控制：1500-2000 字**
- **选择**：比前两章更短（前两章约 2500-2700 字）
- **理由**：第三章是"理论桥梁"，不是"核心概念介绍"；读者在前两章已建立基础，第三章只需精准对接

**5. 内容标注规范：章节末尾统一添加"内容说明"**
- **选择**：在章节末尾用"**本章内容说明**"段落统一标注论文原意、作者分析、工程扩展、虚构示例四类内容
- **理由**：不在正文中频繁打断阅读，保持流畅性；同时满足"标注自我发挥延伸内容"的质量要求
- **标注分类**：
  - **论文原意**：源自 Anthropic《Harness Design for Long-Running Application Development》的内容
  - **作者分析**：基于论文框架的分析推导
  - **工程扩展**：基于论文框架的落地实践补充
  - **虚构示例**：作者构造的说明性场景

## Risks / Trade-offs

**[风险] "Harness 是 DevOps 终极形态"被误解为"Harness 替代 DevOps"**
→ **缓解**：在 C 部分明确说明 Harness 依赖 DevOps 基础设施（CI/CD、代码库、测试环境），强调"站在肩膀上"而非"推倒重来"

**[风险] 反馈环速度的对比缺乏具体数字，显得空泛**
→ **缓解**：使用对比框架（传统：小时级/天级；Harness：分钟级），不编造精确数字，但给出量级差异

**[风险] B 部分（重新诠释的实践）过于简略，读者觉得敷衍**
→ **缓解**：每个例子只用一句话说明变化本质（如"代码审查从'人读代码'变成'Evaluator 运行验证'"），不展开细节；让读者感受到"点到为止是刻意为之"
