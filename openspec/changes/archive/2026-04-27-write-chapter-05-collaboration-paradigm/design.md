## Context

全书前四章已完成：
- 第一章（背景）：建立问题意识——AI 长时开发中的"失忆"与"自信赞扬"问题
- 第二章（Harness 核心）：介绍三角色（Planner/Generator/Evaluator）、验收标准协商、上下文交接
- 第三章（DevOps 基础）：将 Harness 与 DevOps 持续交付理论对接
- 第四章（SRE 基础）：将 Harness 与 SRE 可靠性工程对接，建立质量量化框架

第五章需要完成从"技术架构"到"人机协作范式"的跃迁，回答核心问题：当 AI 接管执行后，人做什么？

## Goals / Non-Goals

**Goals:**
- 建立"验收驱动"的人机协作范式：人类从"执行者"变为"判断者"，保留最终验收权
- 分角色阐述六类岗位（产品、程序员、架构师、项目经理、测试、运维）与 AI 的具体分工边界
- 提出"经验层 + Agent 设计能力"价值放大的核心论点，与第二章的"Agent 范式 mastery"形成呼应
- 用"数控机床与八级钳工"类比让读者直观理解能力门槛重构
- 与前后章形成逻辑闭环：承接第四章的"质量量化"，引出第六章的"闭环框架"

**Non-Goals:**
- 不涉及具体的 Prompt 工程技巧或工具链配置（留给后续实战章节）
- 不展开讨论组织变革管理或岗位裁撤问题（聚焦能力模型而非人事策略）
- 不重复第二章已经详细论述的三角色技术机制

## Decisions

**1. 章节结构采用"论点→分角色论证→类比升华→机制总结"的递进结构**
- 理由：先建立范式认知（5.1），再具体到各角色（5.2），再用类比强化记忆（5.3），最后收束到验收机制（5.4）。这与读者"先懂为什么，再看怎么做"的认知路径一致。
- 替代方案：按角色逐个展开（5.1 产品、5.2 程序员...）——被拒绝，会导致范式主线被角色细节稀释。

**2. 程序员角色的论述突出"Agent 范式 mastery"而非"代码能力"**
- 理由：这是本章最具区分度的观点。初级程序员手动改 AI 代码，高级程序员优化 Agent 模型让 AI 产出更好——这个对比能精准击中当前行业认知痛点。
- 具体展开：不直接改 AI 代码 → 优化上下文、工具链、反馈闭环。

**3. 5.2 各角色采用统一模板："AI 接管什么 → 人类保留什么 → 关键转变"**
- 理由：统一结构便于读者横向比较，也能强化"执行层被接管、判断层被保留"的主线。
- 每个角色的三段式：AI 负责执行/生成/测试...；人类负责判断/权衡/设计...；核心转变是从 X 到 Y。

**4. "数控机床与八级钳工"类比作为 5.3 核心框架**
- 理由：工业史上有大量"执行技能被机器取代、调试/判断技能升值"的先例，类比能让抽象趋势具象化。
- 历史锚点：数控机床普及后，操作工贬值，能编程/调机/诊断的技师升值。

**5. 5.4 验收驱动机制将 Harness Evaluator 映射到人类工作**
- 理由：避免"人类验收"与"AI 验收"两张皮。人类的验收权不是替代 Evaluator，而是在 Evaluator 之上增加业务正确性维度。
- 分层：Evaluator 验"对不对"（功能/性能/安全）→ 人类验"值不值"（业务价值/战略契合/用户体验）。

## Risks / Trade-offs

- **[风险] 角色论述可能变成空洞的"人机分工表"，缺乏洞察力**
  → 缓解：每个角色必须包含一个"反直觉的转变"（如程序员的核心能力不再是写代码），避免泛泛而谈。

- **[风险] "数控机床"类比可能让读者产生"AI 取代人"的焦虑，偏离"价值重构"的主旨**
  → 缓解：明确锚定"八级钳工升级"而非"工人下岗"——强调是人的能力重心转移，不是人被边缘化。

- **[风险] 5.2 六个角色篇幅不均，导致章节臃肿**
  → 缓解：每个角色控制在 400-600 字，重点角色（程序员、架构师）可略长，其余保持简洁。用表格或列表压缩常规内容。

- **[权衡] 5.4 验收机制与第二章人类角色章节有内容重叠**
  → 接受这个重叠，但改变视角：第二章从"Harness 框架需要什么"出发，第五章从"人的工作范式如何重构"出发。前者是功能必要性，后者是认知升级。
