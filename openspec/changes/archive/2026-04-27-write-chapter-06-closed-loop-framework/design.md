## Context

全书前四章已完成理论框架建立（Harness 核心思想、DevOps 基础、SRE 可靠性工程、人机协作范式），第五章完成了"人如何与 AI 共处"的认知升级。当前需要一个"框架大图"将碎片整合为系统，并为第七章角色细化和下篇工程实践奠定坐标系。

## Goals / Non-Goals

**Goals:**
- 建立 Harness 全生命周期闭环框架：四阶段（规划/实现/交付/运营）重叠运转的流水线模型
- 核心论点：Harness 的验收不是"检查清单"，而是"上下文交接"——交接文件的质量决定下一阶段能否零损耗继承
- 论证闭环的驱动力：验收作为阶段间闸口 + 监控数据实时回流触发新一轮迭代
- 与前后章衔接：承接第五章"人机协作范式"，引出第七章"多种专业化角色"

**Non-Goals:**
- 不讲具体角色的能力边界（留给第七章）
- 不讲具体技术实现（AI Agent 研发、看板状态、流程流转等留给第八章及下篇）
- 不复述第四、五章的验收理论（而是从"交接机制"新视角展开）

## Decisions

**1. 6.1 用"接力赛"隐喻替代"瀑布"批判**
- 理由：直接批判瀑布模型是陈旧的开篇方式。用"接力赛——交接棒的质量决定下一棒"更生动，且直接点出"上下文交接"主题。
- 替代方案：从 DevOps 持续交付的流水线讲起 → 被拒绝，会和第三章内容重叠。

**2. 6.2 的管道图展示"重叠运转"而非"串行流程"**
- 理由：关键洞察是 Planner 在 Generator 编码时就已经基于上一轮的运营数据规划下一轮，四阶段是并行流水线。
- 可视化：上下两行管道图，展示两轮迭代的重叠。

**3. 6.3 的核心视角：上下文交接 = 验收机制**
- 理由：这是只属于第六章的独特角度。第四、五章讲了"为什么验收"和"谁验收"，第六章讲"验收的载体是什么"——不是检查清单，而是交接文件。
- 每个交接环节统一模板：交接文件包含什么 → 验收标准（能否零损耗继承）→ 不通过怎么办。

**4. 已知问题清单在 6.3.3 中用 1 个段落展开**
- 理由：这是很有 Harness 特色的实践点（风险透明度），但不宜单独成节以免臃肿。

**5. 6.3.5 强调"实时回流"而非"阶段结束一次性交接"**
- 理由：运营数据持续流入 Planner，Planner 持续产出新 Design——这是 Harness 闭环能持续运转的关键。

## Risks / Trade-offs

- **[风险] "上下文交接"视角可能让读者觉得抽象，难以和具体工作对应**
  → 缓解：每个交接环节都给出具体的交接文件内容示例（Design 文件包含什么、代码交接文件包含什么），让读者能想象出"交接"的具体形态。

- **[风险] 6.3 四个交接环节结构相似，可能导致阅读疲劳**
  → 缓解：每个交接环节突出一个独特洞察——规划→实现强调"Design 的完整性"，实现→交付强调"已知问题清单"，交付→运营强调"可观测性配置"，运营→规划强调"实时性"。

- **[风险] "重叠运转"概念与下篇按阶段展开的实践章节可能产生冲突**
  → 缓解：在 6.2 明确说明"四阶段重叠是 Harness 的理想态，下篇按阶段展开是为了便于工程落地"——理论和实践的视角差异。
