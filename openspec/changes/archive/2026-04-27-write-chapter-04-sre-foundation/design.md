## Context

前三章已完成：
- 第一章（背景）：铺垫 Harness 的工程化落地难题——AI 失忆与自评估偏差
- 第二章（核心）：阐述 Harness 三角色（Planner/Generator/Evaluator）、验收标准协商、轻量流程
- 第三章（DevOps）：将 Harness 置于 DevOps 反馈环中，讨论反馈密度与成本权衡（$9 vs $200，5-15 轮迭代）

第四章需要在前三章基础上引入 SRE（Site Reliability Engineering，站点可靠性工程）视角，形成"可靠性"维度的完整论述。

## Goals / Non-Goals

**Goals:**
- 建立 Harness 与 SRE 理论之间的概念桥梁
- 用 SRE 的框架重新诠释 Harness 的评估轮次和成本权衡
- 说明 Harness 如何与现有 SRE 实践（SLO、可观测性、混沌工程）互补
- 保持叙事风格，专业术语在前、通俗解释在后
- 在章节末尾提供标准"内容说明"，标注论文原意/作者分析/工程扩展

**Non-Goals:**
- 不深入 SRE 的技术实现细节（如 Prometheus 配置、P95/P99 计算）
- 不讨论具体的告警策略或 on-call 轮班制度
- 不创建新的 Hugo 站点配置或文档结构变更

## Decisions

**结构采用"追问式"引入**
- 从第三章结尾自然追问："分钟级的反馈环如果产出的是不可靠的代码，速度还有什么意义？"
- 这个引入方式与第二章"第一章的追问"形成呼应，保持全书结构一致性

**核心概念映射关系（4 个概念）**
1. SLO ↔ Evaluator 验收标准：两者都是"足够好"的定义
2. 错误预算 ↔ token 成本预算：两者都是"允许在一定范围内不完美"的量化约束
3. 可观测性 ↔ Harness 监控反馈：部署后的数据回流是 Harness 反馈环的延伸
4. 混沌工程 ↔ Evaluator 边界测试：验收标准应包含异常场景

**数据呼应策略**
- 主动引用第三章的成本数据（$9/$200、5-15 轮、硬阈值），在 SRE 框架下重新解读
- 避免让读者感觉第四章是独立篇章，而是"同一故事的可靠性视角"

**Mermaid 图表规划**
- 使用 1 个图表：错误预算与 Harness 迭代的关系图（横轴：迭代轮次，纵轴：质量评分，标注平台期和硬阈值停止线）

## Risks / Trade-offs

**概念深度 vs 可读性的平衡**
- SRE 本身是一个庞大的知识体系，本章不能写成 SRE 入门教材
- 缓解：每个 SRE 概念只取与 Harness 直接相关的核心要义，用一句话通俗解释，不过度展开

**避免重复第二章/第三章内容**
- 监控反馈在第三章已经提到，第四章需要提升到"可观测性"的理论高度，而非简单重复
- 缓解：第三章讲"Harness 有监控回流"，第四章讲"为什么监控回流是 SRE 可观测性的自然延伸"

**"内容说明"的标注尺度**
- SRE 理论（SLO/SLI/SLA、错误预算、混沌工程）均为公开的行业实践，非 Harness 论文内容
- 需要清晰区分：SRE 理论本身属于"行业共识"，Harness 与 SRE 的关联分析属于"作者分析"
