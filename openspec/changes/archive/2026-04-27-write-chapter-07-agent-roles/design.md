## Goals / Non-Goals

- 设计"角色协作协议"：多角色之间如何协商、交接、避免冲突
- 上篇理论收官：从问题→框架→理论→系统→角色，完成完整认知闭环
- 与下篇衔接：第七章为第八章"总体架构"和 09-14"阶段实践"提供角色坐标

**Non-Goals:**

- 不复述前六章已详细论述的三角色基本定义和协作机制

## Decisions

**1. 7.1 用"团队扩张"类比引入角色专业化**
- 理由：读者对"小团队→大团队需要分工"有直观认知。类比：初创公司一人全栈 → 大公司需要前端/后端/DBA 分工。Harness 角色同理——简单场景一个通用角色足够，复杂场景需要专业化。
- 替代方案：从"单一角色的能力边界"理论分析引入 → 被拒绝，太抽象。

**2. 每个角色的变体采用"能力光谱"展示**
- 理由：不是简单罗列角色类型，而是展示一个光谱——从"通用型"到"专业型"的渐变。这能帮助读者理解"什么时候需要专业化"。
- 展示方式：表格或列表，每个变体包含"专长领域""适用场景""与通用型的差异"。

**3. Planner 变体聚焦"输入类型"差异**
- 理由：Planner 的核心差异在于"接收什么输入、产出什么 Design"。
- 变体方向：需求分析型（接收模糊需求）、架构规划型（接收技术约束）、安全规划型（接收合规要求）、数据规划型（接收数据模型）。

**4. Generator 变体聚焦"技术栈"差异**
- 理由：Generator 的核心差异在于"用什么技术实现"。
- 变体方向：前端 Generator（React/Vue/Angular）、后端 Generator（Go/Java/Python）、数据库 Generator（SQL/NoSQL）、DevOps Generator（CI/CD/基础设施）。

**5. Evaluator 变体聚焦"验收维度"差异**
- 理由：Evaluator 的核心差异在于"验什么"。
- 变体方向：功能 Evaluator（正确性）、性能 Evaluator（延迟/吞吐量）、安全 Evaluator（漏洞/合规）、业务价值 Evaluator（用户体验/商业价值）。

**6. 7.5 角色协作协议用"交响乐团"隐喻**
- 理由：多角色协作的难点是"谁什么时候上场、如何配合"。交响乐团隐喻直观：指挥（协调者）、第一小提琴（主导）、其他乐器（配合）、乐章切换（阶段过渡）。

## Risks / Trade-offs

- **[风险] 角色变体罗列可能变成"清单式"写作，缺乏洞察力**
  → 缓解：每个变体必须回答"为什么需要这个专业化"——不是因为它存在，而是因为通用型在这个场景下会失败。

- **[风险] 7.2-7.4 三个角色变体结构高度相似，阅读疲劳**
  → 缓解：每个角色的变体展示采用不同侧重——Planner 侧重"输入差异"，Generator 侧重"技术栈差异"，Evaluator 侧重"验收维度差异"。

- **[风险] 角色协作协议（7.5）可能过于抽象，缺乏可操作性**
  → 缓解：用具体场景说明协作协议如何工作（如"前端 Generator 和后端 Generator 如何对接 API 契约"）。

- **[风险] 与下篇 08"总体架构"的内容边界不清**
  → 缓解：明确区分——第七章讲"角色有哪些"，第八章讲"这些角色如何在技术架构中落地"。第七章是能力模型，第八章是系统架构。
