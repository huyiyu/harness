## Goals / Non-Goals

**Goals:**
- 建立 Harness 系统的顶层架构视图：三大核心组件（Lifecycle 服务、AI 服务、DevOps 平台）
- 定义项目生命周期状态机和三个审查点（需求准入、功能提交、生产准入）
- 明确各审查点的审核模式（人工/AI/混合）及可配置性
- 为后续各阶段章节提供统一的架构语境和术语约定
- 保持与上篇理论内容的一致性（角色定义、协作范式、闭环框架）

**Non-Goals:**
- 不涉及具体实现代码
- 不深入某一阶段的详细设计（留给第9-14章）
- 不展开性能优化或安全加固的专项方案
- 不做开源协议的法律分析（仅强调必须研读，避免触碰红线）

## Decisions

**1. 服务拆分：三大核心组件**
- **选择**：Lifecycle 服务（状态机/流程引擎）+ AI 服务（能力提供者）+ DevOps 平台（成熟产品组合）
- **理由**：
  - 避免过度拆分导致运维复杂度爆炸
  - DevOps 组件选用成熟开源产品，不重复造轮子
  - AI 能力统一封装，按需暴露，避免"为了 AI 而 AI"
- **替代方案**：微服务拆分（Planner/Generator/Evaluator 各独立服务）被否决——初期运维成本高，且三者常需共享上下文

**2. Lifecycle 服务：项目生命周期状态机**
- **选择**：Lifecycle 服务作为"指挥者"，内置项目状态机，通过 webhook/API 调用 AI 服务和 DevOps 平台
- **状态机核心流程**：
  ```
  人类立项 → 需求准入审查 → 设计/开发/测试迭代 → 功能提交审查 → 生产准入审查 → 部署上线 → 运维监控 → 评审演进
  ```
- **三个审查点**：
  1. **需求准入**：AI + 人工审核 → 系统完善后可配置为"仅 AI"
  2. **功能提交**（进入多轮测试）：AI + 人工审核 → 系统完善后可配置为"仅 AI"
  3. **生产准入**：始终**人工 + AI 共同审核**，不可配置为纯 AI
- **理由**：
  - 项目由人类发起，体现人类对项目目标的掌控
  - 审查点可配置，支持从"人机协作"到"全自动化"的渐进演进
  - 生产准入保留人工审核，守住安全底线

**3. AI 服务的"按需扩展"原则**
- **选择**：AI 服务作为统一的能力提供者，内部包含 Planner/Generator/Evaluator 模块，但对外暴露为"能力接口"
- **AI 介入决策树**：
  ```
  任务进入系统
      │
      ├─ 结构化、重复性高、有明确验收标准？
      │       ├─ 是 → 规则引擎 / 模板化工具（不用 AI）
      │       └─ 否 → 需要理解上下文、做判断？
      │               ├─ 是 → AI 介入
      │               └─ 否 → 人类主导
      │
      └─ 风险等级：低 → AI 全自动 / 中 → AI+人类确认 / 高 → AI 辅助+人类决策
  ```
- **理由**：明确"不是为了 AI 而 AI"，成熟机制优先，AI 只在需要理解上下文、做判断的场景介入

**4. 核心服务技术栈**
- **选择**：
  | 服务 | 语言 | 理由 |
  |------|------|------|
  | Lifecycle 服务 | Java | 工业化成熟，生态完善，适合构建企业级状态机和流程引擎 |
  | AI 服务 | Python | 生态丰富，易于接入 Claude API 及其他大模型 SDK |
- **理由**：语言选型服务于业务目标——Lifecycle 需要稳定性、成熟度和工业级支持；AI 服务需要快速迭代和丰富的 ML/LLM 生态

**5. DevOps 平台：成熟产品组合**
- **选择**：
  | 功能 | 选型 | 开源协议 | 理由 |
  |------|------|----------|------|
  | 代码管理 | GitLab | 需研读 | API 完善，生态成熟 |
  | CI | GitLab CI | 需研读 | 与 GitLab 天然集成 |
  | 部署 | ArgoCD | 需研读 | GitOps 模式成熟 |
  | 制品库 | Nexus | 需研读 | 容器镜像 + 依赖管理 |
  | 可观测性（短期） | SkyWalking | 需研读 | APM 能力强，部署简单 |
  | 可观测性（长期） | OTel + Prometheus + Grafana | 需研读 | 云原生标准，可扩展 |
  | 消息队列 | RocketMQ | 需研读 | 业务功能强，支持事务消息 |
- **理由**：选用工业化成熟产品，降低初期建设成本，团队专注 Harness 核心差异化能力
- **⚠️ 重要**：所有开源产品选型前必须研读开源协议，评估商业使用风险，避免触碰红线。协议风险是架构决策的必要考量因素

**6. Lifecycle 服务复杂度：领域拆分**
- **选择**：即使初期部署为单一进程，代码层面按领域拆分模块
  - 需求域：需求准入、需求变更管理
  - 开发域：功能提交、多轮测试迭代
  - 部署域：生产准入、部署上线、运维监控
  - 评审域：评审演进、知识沉淀
- **理由**：避免"上帝服务"，为未来真正拆分服务预留结构

**7. MCP 网关：统一的工具接入层**
- **选择**：MCP 网关作为 AI 服务内部的独立模块，所有服务的工具调用统一通过 MCP 网关
- **架构**：
  ```
  ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
  │ Lifecycle 服务  │     │    AI 服务      │     │   DevOps 平台   │
  │    (Java)       │     │   (Python)      │     │                 │
  │                 │     │  ┌───────────┐  │     │  • GitLab       │
  │ • 状态机        │────▶│  │ MCP 网关  │  │────▶│  • ArgoCD       │
  │ • 流程编排      │     │  │           │  │     │  • Jenkins      │
  │ • 审查调度      │     │  │• 协议转换 │  │     │  • SkyWalking   │
  │                 │     │  │• 认证管理 │  │     │  • RocketMQ     │
  └─────────────────┘     │  │• 审计日志 │  │     │                 │
                          │  │• 限流熔断 │  │     │                 │
                          │  └───────────┘  │     │                 │
                          │  • Planner      │     │                 │
                          │  • Generator    │     │                 │
                          │  • Evaluator    │     │                 │
                          └─────────────────┘     └─────────────────┘
  ```
- **统一化理由**：
  - 避免 Java/Python 两套 MCP 实现，降低维护成本
  - MCP 网关专注协议转换和工具适配，是其核心职责
  - Lifecycle 服务通过 HTTP/gRPC 调用 MCP 网关，无需关心底层协议
  - Spring AI 的 MCP 支持尚不成熟，不如 Python 生态完善
  - 工具对接集中管理，便于版本升级、故障隔离、安全审计
- **例外**：Lifecycle 服务的简单 webhook 回调和状态查询可直接使用 HTTP client，无需经过 MCP 网关

**8. 通信机制**
- **服务间通信**：gRPC（Lifecycle ↔ AI 服务，结构化、低延迟）
- **工具调用**：MCP 协议（AI 服务 ↔ DevOps 平台，标准化工具接入）
- **事件通知**：Webhook / GitHook（DevOps 平台 → Lifecycle 服务，异步状态更新）
- **消息队列**：RocketMQ（异步任务、事件驱动、削峰填谷）
- **理由**：成熟、标准化、被广泛支持，不为了"智能化"而引入不必要的复杂度

## Risks / Trade-offs

- **[风险]** Lifecycle 服务成为单点瓶颈 → **缓解**：领域拆分 + 事件驱动架构，核心状态机轻量，具体动作由事件消费者执行
- **[风险]** 审查点配置为"仅 AI"后，质量失控 → **缓解**：保留审计日志和回退机制，AI 决策可回溯、可审查
- **[风险]** 开源协议合规风险 → **缓解**：所有开源产品选型前必须研读协议，评估商业使用风险；建立协议审查清单，高风险组件准备替代方案（如 Nexus 可替换为 Harbor）
- **[风险]** SkyWalking 到 OTel 的迁移成本 → **缓解**：短期 SkyWalking 做 APM，长期引入 OTel 做指标/日志标准化，两者可共存
- **[风险]** 与上篇理论重复 → **缓解**：聚焦"如何落地为系统"，每个概念指向系统化的实现路径
- **[权衡]** 架构的完整性与简洁性 → **选择** 简洁优先，只保留核心模块，扩展模块留给后续章节
