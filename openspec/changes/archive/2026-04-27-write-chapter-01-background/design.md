## Decisions

**1. 叙事结构：故事 A（单个项目困境）+ 故事 C（规模化困境）的组合**
- **选择**：先讲"单个项目为什么会出问题"（交付延期、质量幻觉），再讲"如果不解决，规模化时会发生什么"（组织能力瓶颈）
- **理由**：决策者更容易从"点"到"面"理解问题深度；单个故事有情感代入，规模化视角有战略高度
- **替代方案**：纯历史演进型（从 LLM Agent 发展阶段讲起）——被认为对决策者不够直接，技术味过重

**2. 案例来源：Claude Code 公开实践 + Anthropic 团队实测数据**
- **选择**：引用 Anthropic 工程团队公开分享的 Harness 设计经验（Claude Sonnet 4.5 的上下文焦虑、自评估偏差等），以及 Claude Code 在长任务中的典型表现
- **理由**：一手数据比虚构故事更有说服力；与本书核心主题（Anthropic Harness 论文）直接呼应
- **替代方案**：完全虚构故事——被认为缺乏可信度和专业感

**3. 风格：纯叙事，零技术术语**
- **选择**：用"失忆""自夸"等比喻替代"上下文窗口""token 压缩""评估器调优"等技术概念
- **理由**：目标读者为管理层/决策者，需要业务语言而非工程语言
- **替代方案**：技术科普风格——与目标读者画像不符

**4. 章节长度：1500-2500 中文字符**
- **选择**：保持精炼，控制在单屏可读完的范围
- **理由**：入口章节过长会导致读者流失；问题意识建立后应及时收束，把好奇心引向第二章

## Risks / Trade-offs

**[风险] 叙事过于笼统，决策者无法感知真实痛感**
→ **缓解**：使用具体数字锚定（"第 3 天""第 8 天""额外 40% 时间"），引用 Anthropic 团队第一手观察而非泛泛而谈

**[风险] Claude Code 案例过于具体，限制本书适用范围**
→ **缓解**：将案例定位为"一个典型的长时开发场景"，而非"只有 Claude Code 才有这个问题"；强调这是"长时运行 LLM Agent 的结构性问题"

**[风险] 只提问题不讲原理，读者可能感到焦虑而无出口**
→ **缓解**：在章节末尾明确给出"Harness 是解药"的信号（仅命名），并在语气上保持"问题已被识别，解决方案在下一章"的期待感，而非纯负面渲染
